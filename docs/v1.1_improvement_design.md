## 개선 1: 파일 로그 시스템

### 1.1 설계 목적

화면 출력(rich)과 독립적으로, 모든 실행 내역을 파일에 기록하는 로그 시스템.
normal 모드로 실행하더라도 verbose 수준의 상세 정보가 파일에 남아야 한다.

### 1.2 아키텍처

```
사용자 터미널                    로그 파일
─────────────                   ──────────
rich Output 클래스               Python logging 모듈
  ├── normal 모드: 요약만          ├── 항상 DEBUG 레벨로 기록
  ├── verbose 모드: 상세           ├── 출력 모드와 무관
  └── quiet 모드: 에러만           └── 모든 단계의 타이밍 포함
```

두 시스템은 독립적으로 동작한다. Output 클래스의 각 메서드가 호출될 때, 동시에 logging 모듈로도 기록한다.

### 1.3 로그 파일 위치 및 이름

```
logs/yt-excel_2026-03-06_143021.log
```

- 디렉토리: 프로젝트 루트의 `logs/`
- 파일명: `yt-excel_YYYY-MM-DD_HHMMSS.log` (실행 시각 기준)
- `logs/` 디렉토리는 없으면 자동 생성
- `logs/` 디렉토리는 `.gitignore`에 추가

### 1.5 로그에 포함할 타이밍 정보

| 단계 | 기록 항목 |
|------|-----------|
| 전체 파이프라인 | 총 소요 시간 |
| 메타데이터 조회 | 소요 시간 |

### 1.6 로그 레벨 매핑

| logging 레벨 | 용도 |
|--------------|------|
| DEBUG | 세그먼트별 상세 (번역 전/후 텍스트, API 응답 원문) |
| INFO | 단계별 요약, 타이밍, 배치 완료 |
| WARNING | retry 발생, 부분 실패, 배열 길이 불일치 |
| ERROR | 치명적 오류, 파이프라인 종료 |

기본 파일 로그 레벨은 DEBUG다. 모든 것을 기록한다.

### 1.7 보안

- 로그 파일에 API 키를 절대 기록하지 않는다.
- 로그에 "API key validated" 정도만 남기고, 키 값 자체는 포함하지 않는다.

### 1.8 config.yaml 확장

```yaml
# 로그 설정
logging:
  enabled: true                 # 파일 로깅 활성화 (기본: true)
  dir: "./logs"                 # 로그 디렉토리 경로
  level: "DEBUG"                # 파일 로그 레벨
```

### 1.9 구현 범위

- `src/yt_excel/logger.py` — 신규 모듈 (logging 설정, 파일 핸들러)
- `src/yt_excel/cli.py` — Output 클래스에 logging 연동 추가
- `src/yt_excel/translator.py` — 배치별 타이밍 로깅 추가
- `config.yaml` — logging 섹션 추가
- `.gitignore` — `logs/` 추가

## 개선 2: 비동기 번역 (Async Translation)

### 2.1 설계 목적

번역 API 호출을 동시 실행하여 총 번역 시간을 단축한다.

### 2.2 현재 구조 (동기)

```
Batch 1 ──53s──→ Batch 2 ──55s──→ ... Batch 9 ──48s──→ 완료
                                                        총: ~476s
```

9개 배치를 순차 실행. 총 시간 = 모든 배치 시간의 합.

### 2.3 목표 구조 (비동기)

```
Batch 1 ──53s──→
Batch 2 ──55s──→   (동시 최대 3개)
Batch 3 ──50s──→
                   → Batch 4~6 → Batch 7~9 → 완료
                                              총: ~160s (3라운드)
```

동시 실행 수를 제한하여 rate limit을 회피하면서 속도를 개선한다.

### 2.4 동시성 제어

| 파라미터 | 값 | 설명 |
|----------|-----|------|
| `max_concurrent_batches` | 3 | 동시 API 호출 최대 수 |
| `request_interval_ms` | 200 | 동일 유지 (배치 시작 간 최소 간격) |

`asyncio.Semaphore(3)`으로 동시 요청 수를 제한한다.

### 2.5 예상 성능 개선

82 세그먼트, batch_size=10, 9개 배치 기준:

| 구성 | 라운드 수 | 예상 시간 |
|------|-----------|-----------|
| 동기 (현재) | 9 | ~476s |
| 동시 3개 | 3 | ~160s |
| 동시 5개 | 2 | ~110s |

동시 3개만으로도 약 3배 개선.

### 2.6 API 클라이언트 변경

```
현재: openai.OpenAI() → client.chat.completions.create()
변경: openai.AsyncOpenAI() → await client.chat.completions.create()
```

OpenAI Python SDK는 `AsyncOpenAI` 클라이언트를 기본 제공한다. 별도 의존성 추가 없음.

### 2.7 Rate Limit 대응

- `asyncio.Semaphore`로 동시 요청 수 제한
- 429 응답 시 해당 배치만 `Retry-After` 대기 후 재시도 (다른 배치에 영향 없음)
- 모든 배치 시작 전 `request_interval_ms` 간격 유지 (`asyncio.sleep`)

### 2.8 Progress Bar 호환

현재 `rich.progress.Progress`는 동기 환경에서 동작한다.
비동기 배치가 완료될 때마다 `progress.advance()`를 호출하는 방식은 동일하게 유지 가능하다.
다만 여러 배치가 동시에 완료될 수 있으므로, advance 호출은 thread-safe해야 한다.
`rich`의 Progress는 내부적으로 thread-safe하므로 추가 처리 불필요.

### 2.9 결과 순서 보장

비동기 실행 시 배치 완료 순서가 요청 순서와 다를 수 있다.
각 배치에 index를 부여하고, 모든 배치 완료 후 index 순서로 결과를 정렬하여 세그먼트에 매핑한다.

```
배치 생성: [Batch 0, Batch 1, ..., Batch 8] (index 부여)
비동기 실행: 완료 순서 무관
결과 수집: index 기준 정렬 후 세그먼트에 korean 할당
```

### 2.10 실패 처리

- 개별 배치의 retry는 기존과 동일 (최대 3회, exponential backoff)
- 한 배치의 실패가 다른 배치에 영향을 주지 않는다
- 모든 배치 완료 후 실패 배치의 세그먼트는 korean 빈 문자열

### 2.11 동기 모드 유지 (fallback)

asyncio가 문제를 일으킬 경우를 대비해 동기 모드를 config로 선택 가능하게 한다.

```yaml
# config.yaml 확장
translation:
  async_enabled: true             # 비동기 번역 활성화 (기본: true)
  max_concurrent_batches: 3       # 동시 배치 수
```

`async_enabled: false`이면 기존 동기 실행으로 fallback.

### 2.12 구현 범위

- `src/yt_excel/translator.py` — `AsyncOpenAI` 클라이언트, `translate_segments_async()` 추가
- `src/yt_excel/cli.py` — 비동기/동기 분기, `asyncio.run()` 호출
- `config.yaml` — `async_enabled`, `max_concurrent_batches` 추가
- `tests/test_translator.py` — 비동기 번역 테스트 추가

### 2.13 기존 동기 코드와의 관계

기존 `translate_segments()` 함수는 삭제하지 않는다.
`translate_segments_async()`를 새로 추가하고, `cli.py`에서 config에 따라 분기한다.

```
if config.translation.async_enabled:
    results = asyncio.run(translate_segments_async(segments, config))
else:
    results = translate_segments(segments, config)
```

## 구현 순서

```
1. 로그 시스템 (개선 1)
   → 기존 코드 변경 최소, 바로 디버깅에 활용 가능
   → 비동기 번역 개발 시 로그로 성능 측정 가능

2. 비동기 번역 (개선 2)
   → 로그 시스템이 있는 상태에서 개발
   → 배치별 타이밍 비교로 개선 효과 즉시 측정
```

## 브랜치 전략

```
main (안정)
  └── dev (통합)
        ├── feat/logging-system     → 개선 1 개발 → dev에 merge → 검증
        └── feat/async-translation  → 개선 2 개발 → dev에 merge → 검증
                                                         ↓
                                                    dev → main merge
```

각 기능 브랜치에서 기능 단위 커밋을 진행하고, dev에서 통합 테스트를 통과한 후에만 main에 merge한다.
