# 📘 yt-excel v1.2 개선 설계 문서

*v1.1 실사용 피드백 기반 — Progress bar 실시간 업데이트 + 영상 길이 확인*

---

## 배경

v1.1 실사용에서 발견된 두 가지 문제:

1. **[UX-BUG] Progress bar 일괄 업데이트**: 비동기 번역 중 ~170초 동안 progress bar가 0%에서 멈춘 뒤 한꺼번에 100%로 점프. `asyncio.gather()`가 모든 태스크 완료 후에야 반환하기 때문.

2. **[UX] 긴 영상 실수 방지 없음**: 1시간 30분 영상 URL을 입력해도 확인 없이 번역이 진행되어 의도치 않은 API 비용 소비. TED-Ed 기준 5~10분이 일반적이지만, 도구 자체는 범용으로 사용 가능해야 한다.

---

## 개선 1: Progress bar 실시간 업데이트

### 1.1 문제 원인

현재 `translate_segments_async()` 내부:

```
tasks = [translate_batch_with_retry_async(...) for batch in batches]
results = await asyncio.gather(*tasks)
# gather() 반환 후에야 on_batch_complete 콜백 호출
for result in results:
    on_batch_complete(...)
```

`asyncio.gather()`는 모든 태스크가 끝나야 반환하므로, 개별 배치 완료 시점에 콜백이 실행되지 않는다.

### 1.2 수정 방안

`asyncio.gather()` 대신 `asyncio.as_completed()` 패턴으로 전환한다.

```
tasks = [translate_batch_with_retry_async(...) for batch in batches]
results = []
for coro in asyncio.as_completed(tasks):
    result = await coro
    results.append(result)
    on_batch_complete(batch_size)  # 즉시 호출 → progress bar 실시간 업데이트
results.sort(key=lambda r: r[0])  # index 기준 정렬 (순서 보장)
```

### 1.3 동작 변경

| 항목 | 변경 전 (gather) | 변경 후 (as_completed) |
|------|-------------------|------------------------|
| 콜백 시점 | 모든 배치 완료 후 일괄 | 각 배치 완료 즉시 |
| progress bar | 0% → (멈춤) → 100% 점프 | 0% → 11% → 22% → ... 점진적 증가 |
| 결과 순서 | gather가 입력 순서 보장 | 완료 순서로 반환 → index 정렬 필요 |
| Semaphore 동작 | 동일 | 동일 (태스크 생성 시 이미 적용) |

### 1.4 결과 순서 보장

`as_completed()`는 완료 순서로 반환하므로, 기존의 index 기반 정렬 로직이 더 중요해진다. 현재 구현에 이미 `results.sort(key=lambda r: r[0])` 이 있으므로 추가 변경 불필요.

### 1.5 영향 범위

- `src/yt_excel/translator.py` — `translate_segments_async()` 함수 내부만 변경
- 다른 모듈 변경 없음
- 기존 테스트에서 결과 순서 보장 테스트(`test_result_order_preserved_across_batches`)가 이미 있으므로, 리팩토링 후에도 동일하게 통과해야 한다

### 1.6 테스트

- 기존 비동기 테스트 전체 통과 확인 (regression 방지)
- 추가 테스트: `on_batch_complete` 콜백이 각 배치 완료 시점에 즉시 호출되는지 검증 (호출 순서가 완료 순서와 일치하는지)

## 개선 2: 영상 길이 확인 (Duration Threshold)

### 2.1 설계 목적

긴 영상에 대한 실수 방지. 번역을 차단하는 것이 아니라, 사용자에게 결정권을 준다.

### 2.2 동작 흐름

```
Video Metadata Fetcher 완료
→ duration이 threshold 이하 → 정상 진행 (확인 없음)
→ duration이 threshold 초과:
    ⚠ This video is 1h 30m 00s long (threshold: 15m).
    Estimated segments: ~900
    Estimated cost: ~$0.010

    Continue translation? [y/N]:
    → y → 정상 진행
    → N 또는 빈 입력 → 종료
```

기본 응답은 N (안전 방향). 사용자가 명시적으로 y를 입력해야만 진행한다.

### 2.3 config.yaml 확장

```yaml
# 필터링 설정
filter:
  min_duration_sec: 0.5
  min_text_length: 2
  max_duration_minutes: 15        # [NEW] 확인 없이 진행 가능한 최대 영상 길이 (분)
                                  # 0 = 확인 비활성화 (모든 영상 무조건 진행)
```

- 기본값: 15분 (TED-Ed 기준 넉넉한 상한)
- 0으로 설정하면 길이 확인을 비활성화 (모든 영상 무조건 진행)

### 2.4 CLI 옵션 확장

```
--yes, -y    확인 프롬프트를 건너뛰고 항상 진행
```

배치 스크립트나 자동화에서 사용. `--yes` 플래그가 있으면 threshold를 초과해도 확인 없이 진행한다.

### 2.5 출력 모드별 동작

| 모드 | threshold 초과 시 동작 |
|------|------------------------|
| normal | 확인 프롬프트 표시, 사용자 입력 대기 |
| verbose | 동일 (추가로 상세 비용 정보 표시) |
| quiet | 확인 프롬프트를 띄울 수 없으므로 자동 종료 + 에러 메시지 |
| `--yes` 플래그 | 확인 없이 진행 (모든 모드에서 동일) |
| `--dry-run` | threshold 초과여도 확인 없이 분석 진행 (API 호출 없으므로 비용 없음) |

### 2.6 확인 프롬프트 형식

```
⚠ This video is 1h 30m 00s long (threshold: 15m).
   Estimated segments: ~900 (based on 10 seg/min average)
   Estimated cost: ~$0.010 (gpt-5-nano)

   Continue translation? [y/N]:
```

- 영상 길이: `HH:MM:SS` 대신 읽기 쉬운 `1h 30m 00s` 형식
- 예상 세그먼트: 분당 10개 세그먼트로 추정 (실제 VTT 다운로드 전이므로 추정치)
- 예상 비용: 추정 세그먼트 × 모델별 토큰 단가
- threshold 값도 함께 표시하여 왜 확인이 뜨는지 알 수 있게

### 2.7 확인 시점

파이프라인 흐름에서 **Video Metadata Fetcher 직후, Caption Lister 이전**에 수행한다.

```
Video Metadata Fetcher ← retry 3회
  ↓
Duration Check (threshold 초과 시 확인 프롬프트)   ← [NEW]
  ↓
Caption Lister ← retry 3회
```

메타데이터에서 duration을 얻은 직후에 확인하는 이유:
- 자막 다운로드 전이므로 네트워크 비용 최소화
- Excel 초기화 전이므로 불필요한 파일 생성 방지
- 비용이 발생하는 모든 단계(자막 다운로드, 번역) 이전

### 2.8 Duration 파싱

현재 `fetch_metadata()`가 duration을 `HH:MM:SS` 문자열로 반환한다.
threshold 비교를 위해 초 단위로 변환하는 유틸리티가 필요하다.

```
"00:05:19" → 319초 → 5.3분 → threshold(15분) 이하 → 통과
"01:30:00" → 5400초 → 90분 → threshold(15분) 초과 → 확인 프롬프트
```

### 2.9 로그 기록

- threshold 초과 시: `[INFO] Duration check: 90.0min exceeds threshold 15min, prompting user`
- 사용자 응답: `[INFO] User confirmed: proceeding with long video` 또는 `[INFO] User declined: aborting`
- threshold 이하 시: `[DEBUG] Duration check: 5.3min within threshold 15min`
- `--yes` 플래그 시: `[INFO] Duration check: 90.0min exceeds threshold 15min, auto-confirmed (--yes flag)`

### 2.10 구현 범위

- `src/yt_excel/config.py` — `FilterConfig`에 `max_duration_minutes` 필드 추가
- `src/yt_excel/cli.py` — Duration check 단계 추가, `--yes` 옵션 추가, 확인 프롬프트 구현
- `config.yaml` — `max_duration_minutes: 15` 추가
- `tests/` — duration check 테스트 (threshold 이하/초과, --yes, --quiet, --dry-run, 값 0)

## 브랜치 전략

v1.1에서 브랜치 전략이 지켜지지 않았으므로, 이번에는 **기능별 브랜치를 반드시 분리**한다.

```
main (안정)
  └── dev (통합, v1.1 merge 완료 상태)
        ├── fix/progress-bar-realtime    → 개선 1 → dev merge → 검증
        └── feat/duration-threshold      → 개선 2 → dev merge → 검증
                                                        ↓
                                                   dev → main merge
```

**프롬프트에 브랜치 생성/merge 지시를 반드시 포함한다.** 설계 문서의 브랜치 전략을 에이전트가 자율적으로 참조한다고 가정하지 않는다. (v1.1 교훈)
